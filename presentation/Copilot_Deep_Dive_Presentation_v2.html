GitHub Copilot CLI

Full agentic AI in the terminal — the direct comparison point to Claude Code

Execution Modes: Plan vs. Autopilot

Three interactive modes plus a programmatic interface. Cycle between modes with Shift+Tab during a session.

Ask / Execute

Default

The default mode. Copilot responds to prompts directly, asks for tool approval step-by-step, and executes commands one at a time. Best for exploratory work and learning the codebase.

$ copilot
# Start interactive session
# Copilot asks before each action
Step-by-step Approval required

Plan Mode

Structured

Copilot analyzes the request, asks clarifying questions, and builds a structured implementation plan before writing any code. Catches misunderstandings early. Keeps you in control of complex, multi-step tasks.

# Press Shift+Tab to enter Plan mode
# Or select /plan in the session
# Copilot outputs a plan, then executes
Plan first Clarifying Qs Best for complex tasks

Autopilot Mode

Experimental

Encourages the agent to continue working until the task is completed without pausing for intermediate user input. Requires --experimental flag. Closest to Claude Code’s autonomous loop.

$ copilot --experimental
# Then toggle to Autopilot with Shift+Tab
# Agent works until task is done
Autonomous Experimental

Programmatic & Headless Interface

Pass a single prompt on the command line for scripting and CI integration:

# Single prompt — executes and exits
$ copilot -p "Show me this week's commits and summarize" \
  --allow-tool='shell(git)'
# Fully headless — no user prompts
$ copilot --allow-all-tools -p "Kill the process on port 3000"

Piped input from scripts or other commands:

# Pipe output into Copilot
$ ./generate-options.sh | copilot
# Combine with fleet for parallel work
$ copilot -p "/fleet Refactor auth, tests, and docs" \
  --no-ask-user

For Claude Code users: Autopilot mode is the closest equivalent to Claude Code’s default autonomous behavior. Plan mode is unique to Copilot CLI — it helps teams enforce “think before you code” discipline that your governance model requires.

Agent Environment: Context, Tools, and Files

What the Copilot CLI agent can see, access, and interact with inside your workspace.

📁

Workspace Context

Operates in the current working directory. Reads files, project structure, dependencies, and documentation. Trusted directory scoping on first launch. Add more directories with /add-dir.

🛠️

Built-in Tools

Shell commands (git, npm, pip, curl, etc.), file read/write, GitHub MCP server (issues, PRs, Actions), LSP servers for go-to-definition and diagnostics, and image analysis.

🔌

Custom MCP Servers

Connect to internal APIs, databases, and DevOps systems via MCP. Manage with /mcp slash command. Same protocol as Claude Code — build once, use everywhere.

🧠

Auto-Compaction & Infinite Sessions

At 95% of the token limit, Copilot automatically compresses conversation history in the background without interrupting your work. Enables virtually infinite sessions. Manual compaction available with /compact. View token usage with /context.

🧰

LSP Integration

Configure Language Server Protocol servers for enhanced code intelligence — go-to-definition, hover docs, diagnostics. Set up via ~/.copilot/lsp-config.json (user) or .github/lsp.json (repo). Gives the agent IDE-grade understanding in the terminal.

Key differentiator vs. Claude Code: The built-in GitHub MCP server gives Copilot CLI native access to your GitHub data — issues, PRs, Actions, code search. Claude Code requires manual MCP server setup for equivalent functionality.

Fleet & Sessions: Parallel Execution at Scale

/fleet decomposes tasks into parallel sub-agents. /delegate dispatches work to the cloud coding agent for a PR. Sessions are persistent with metrics.

🚀

/fleet — Parallel Multi-Agent Execution

Dispatch multiple sub-agents in parallel. An orchestrator decomposes tasks, identifies parallelism, manages execution waves, and synthesizes results.

# Break a big task into parallel tracks
> /fleet Refactor the auth module, update all tests, and rewrite docs/auth/

How it works:

  1. Decomposes your task into discrete work items with dependencies
  2. Identifies which items can run in parallel vs. must serialize
  3. Dispatches independent items as background sub-agents
  4. Polls for completion, dispatches next wave
  5. Verifies outputs and synthesizes final artifacts
New — April 2026 Parallel agents
📤

/delegate — Background Coding Agent

Dispatch the Copilot coding agent to work in the background and open a PR. Works asynchronously — you keep working while Copilot creates the branch, codes, tests, and opens the PR.

> /delegate Fix issue #42 and use playwright to verify
📊

Session Management

Every session has an ID, metrics, and can be exported. Track premium requests, duration, and code changes.

/sessionView session metrics (requests, duration, lines changed)
/shareExport session as markdown or GitHub Gist
/tasksView running background tasks from /fleet or /delegate
/clearReset session for a new task
/compactManual context compression

Fleet best practices: Assign each sub-agent distinct files — there’s no file locking, so the last writer wins. Declare dependencies explicitly. Use /tasks to monitor progress. Reference custom agents for specialized tracks: /fleet Use @docs-writer.md for docs and default for code.

Permissions & Boundaries

Fine-grained control over what the agent can and cannot do. Enterprises can enforce strict boundaries per team.

Tool Approval Flow

When Copilot needs a tool that could modify or execute, it presents three options:

1️⃣ Yes (once) Allow this specific command, this time only
2️⃣ Yes (session) Approve this tool type for the entire session
3️⃣ No + feedback Reject and tell Copilot what to do instead (inline steering)

Command-Line Granular Controls

# Allow everything except destructive commands
$ copilot --allow-all-tools \
  --deny-tool='shell(rm)' \
  --deny-tool='shell(git push)'
# Only allow git reads and file writes
$ copilot --allow-tool='shell(git log)' \
  --allow-tool='shell(git diff)' \
  --allow-tool='write'
Spec SyntaxWhat It Controls
shell(COMMAND)A specific shell command (e.g., rm, curl)
shell(git SUBCOMMAND)A specific git subcommand (e.g., git push)
shellAll shell commands
writeFile modifications (non-shell)
MCP_SERVER(tool)Specific MCP server tool

--deny-tool always takes precedence over --allow-tool.

Agent Mode in VS Code

The primary interaction model. An autonomous AI assistant that plans, edits multi-file, runs commands, and self-corrects — all inside your IDE.

The Agent Loop

1. Describe your goal

Natural language description of what you want to build or fix

2. Agent plans the approach

Infers required subtasks, identifies files to change, and maps dependencies

3. Multi-file editing

Edits files across the workspace — creates, modifies, deletes

4. Terminal commands

Installs dependencies, runs builds, executes tests

5. Self-correction

Catches errors, fixes them, iterates until all subtasks pass

6. Review & approve

Inline diffs with keep/undo per change, automatic checkpoints for rollback

Four Agent Runtime Environments

Local

Runs on your machine

Interactive in VS Code. Full workspace access, tools, models. Approve changes in real-time.

Copilot CLI

Runs locally, background

Background execution on your machine. Optional Git worktree isolation. Reports back when done.

Cloud

GitHub infrastructure

Remote sandbox via GitHub Actions. Creates branches and PRs. Team collaboration.

Third-Party

Anthropic / OpenAI SDK

Uses provider’s agent harness (Claude Agent SDK, OpenAI Codex). Local or cloud.

Permission Levels

🛡️ Default Read-only tools auto-approved; write actions need confirmation
⚡ Bypass Auto-approve all tool calls; agent may still ask clarifying questions
🤖 Autopilot Full autonomy — auto-approves, auto-responds, works until done

Session handoff: Users can seamlessly hand off a session from one agent type to another (e.g., Local → CLI → Cloud). VS Code carries over the full conversation history and context.

Model Enablement & Governance

Three-tier policy cascade from Enterprise → Organization → Repository. Centralized control over models, features, privacy, agents, and MCP.

Enterprise Level

Enforces or delegates. Enterprise owners define policies that cascade to all orgs. Can lock policies (no override at org level) or set “No policy” to let orgs decide.

  • AI Controls tab (central control plane)
  • Agents, Copilot, MCP sidebar sections
  • Content exclusions
  • Model allow/deny lists

Organization Level

Configures within enterprise boundaries. Org owners set policies via Copilot settings. Options: Enabled, Disabled, or Unconfigured.

  • Feature policies (Chat, Agent, Code Review)
  • Privacy policies (public code matching)
  • Model availability per org
  • MCP allowlist configuration

Repository Level

Inherits and extends. Repo admins can add content exclusions and configure automatic code review rulesets.

  • Content exclusions (path patterns)
  • Automatic review rulesets
  • Custom instructions
  • AGENTS.md / copilot-instructions.md

Policy Types

Policy TypeWhat It ControlsOptions
Feature Policies Availability of Chat in IDE, Code Review, Coding Agent, Agent Mode, CLI Enabled / Disabled / Let orgs decide
Privacy Policies Suggestions matching public code, data handling Allowed / Blocked
Model Policies Which AI models are available beyond basic tier Per-model enable/disable
Agent Policies Coding agent, third-party agents (Claude, Codex), custom agents Enabled everywhere / Let orgs decide
MCP Policies MCP server access (disabled by default for Enterprise) Allow all / Registry allowlist / Disabled

For governance teams: This is the policy infrastructure you were planning to build from scratch for Claude Code. With Copilot Enterprise, it’s already built. Enterprise admins lock down models, agents, and MCP access from a single AI Controls dashboard. No custom RBAC pipelines needed.

GitHub Agentic Workflows: Continuous AI for Your Repository

Write automation in markdown, compile to GitHub Actions, and run coding agents (Copilot, Claude, Codex) on a schedule or in response to events — with defense-in-depth security. Developed by GitHub Next & Microsoft Research.

How Agentic Workflows Work

1. Write a Markdown workflow

Define triggers, permissions, and safe outputs in YAML frontmatter. Write natural-language task instructions in the markdown body.

2. Compile with gh aw compile

The CLI compiles your .md into a hardened .lock.yml GitHub Actions workflow with SHA-pinned actions, schema validation, and security scanning.

3. Trigger fires

Schedules (daily, hourly), GitHub events (issue opened, PR created), slash commands, label changes, or manual dispatch.

4. Sandboxed AI agent runs

A coding agent (Copilot, Claude, or Codex) runs inside an isolated container with read-only permissions and a network firewall.

5. Safe outputs applied

The agent produces structured artifacts. A separate job with scoped write permissions applies only permitted operations (issues, comments, PRs, labels).

6. Threat detection & delivery

An AI-powered threat scan checks for prompt injection, leaked credentials, and malicious patterns before any output reaches your repository.

Example: Daily Status Report Workflow

---
on:
  schedule: daily
permissions:
  contents: read
  issues: read
  pull-requests: read
safe-outputs:
  create-issue:
    title-prefix: "[team-status] "
    labels: [report, daily-status]
    close-older-issues: true
---

## Daily Issues Report

GitHub issue.

## What to include

Workflows are .md files stored in .github/workflows/. The YAML frontmatter configures what the agent can do; the markdown body tells it what to do.

# Recompile after editing frontmatter

Defense-in-Depth Security

Five security layers work together to contain the impact of a confused or compromised agent:

🔒Read-only tokensAgent receives a GitHub token scoped to read-only. Even if it tries to push code, the token won’t allow it.
🚫Zero secrets in agentWrite tokens, API keys, and credentials are never passed to the agent process. Nothing to steal.
🧱Containerized + FirewallAgent runs in an isolated container. The AWF (Agent Workflow Firewall) enforces a domain allowlist via Squid proxy — all other traffic is dropped.
📤Safe outputsAgent produces structured artifacts describing intended actions. A separate job with scoped write permissions applies only what the workflow permits.
🔍Threat detectionAI-powered scan checks for prompt injection, leaked credentials, and malicious code before any output reaches the repo.

How Premium Requests Work

Each model has a multiplier that determines how many premium requests a single interaction consumes:

  • 3x (Premium): Opus 4.5, Opus 4.6 — deep reasoning tasks
  • 30x: Opus 4.6 fast mode — preview, for latency-critical use
